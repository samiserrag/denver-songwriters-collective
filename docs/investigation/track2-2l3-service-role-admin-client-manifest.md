This is the maintained Track 2 manifest for privileged Supabase access reachable
from Track 2 routes, workers, and helpers. It is not a remediation PR and does
not enable any new runtime behavior. Future 2D, 2F, 2J, and 2K implementation
PRs must update this manifest before adding or changing any
`createServiceRoleClient()`, `getServiceRoleClient()`, direct service-role key,
or `auth.admin` usage in guarded Track 2 surfaces.

## Maintenance Rules

For every manifest entry, keep these fields current:

- file
- route/helper
- resource family
- actor check
- object/resource check
- tables or auth resources touched
- read/write purpose
- audit/logging expectation
- negative tests present or missing
- status

Service-role clients bypass RLS. Do not rely on Supabase RLS as the primary
authorization boundary for any row that uses service-role or auth-admin access.
The caller must establish the actor and object/resource scope before privileged
reads or writes, unless the row is explicitly a public-safe read exception.

The source-check scaffold intentionally scans explicit Track 2 roots and helper
files rather than the entire repository. If a future Track 2 PR adds a new route,
worker, helper, or server action with privileged Supabase usage, update both the
manifest and the guarded root or sentinel list in the same PR.

## Out-Of-Scope But Known Privileged Usage

The source-check scaffold is intentionally Track 2-scoped. It does not scan every
existing service-role/admin-client usage in the repository. Current known
out-of-scope examples include gallery collaborator routes, Friends of the
Collective pages, digest routes, guest verification routes, and admin user server
actions. If a future Track 2 PR depends on one of those paths, move that file
into the guarded roots and add a current manifest row in the same PR.
